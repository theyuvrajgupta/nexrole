# Career Suite: Hosted App Build Brief

This brief is the starting point for building the hosted version of the JD Career Suite in Claude Code. It carries over the design decisions, working prompts, and hard-won lessons from the prototype phase (a Claude Skill plus a standalone HTML artifact, both included in this repo as reference material). Read this fully before scaffolding.

## What this app is

A job application suite that takes a candidate from "what roles should I apply to" through "I am ready for the interview." Four connected stages: a job finder that surfaces real current openings matched to the candidate's profile, a fit analyzer that scores a pasted job description against their full career history with honest gap identification, a document generator that produces tailored resumes and cover letters in selectable formats, and an interview prep builder with distinct structures per interview type (consulting, product management, investment banking, VC/PE, tech leadership, general behavioral).

The founding principle, which must survive every feature decision: the app never fabricates experience. Reframing real experience in a job description's vocabulary is allowed when honest. Inventing tools, titles, or outcomes is not. Every generation prompt carries this rule, and it is why the fit analyzer exists at all: an inflated fit score wastes interview rounds, so the analyzer is built to be honest, including telling the user to skip a role.

## Who it is for at launch

The owner (Yuvraj), his wife, and a small circle of MBA classmates. This means invite-based access (a simple allowlist or invite code is enough, no public signup flow, no billing), but real authentication and strict per-user data isolation, because users store their own resumes and application history. Assume roughly 10 to 30 users at launch. Do not build for scale that does not exist yet, but do not cut corners on data isolation.

## Recommended stack

Next.js (App Router) deployed on Vercel, with Postgres for persistence (Neon or Supabase both fine; if Supabase, its auth can cover the invite-based login too). Anthropic API called exclusively from server routes using the official SDK, model claude-sonnet-4-5 or newer Sonnet, with the web search tool enabled for the job finder route. The Anthropic API key lives in server environment variables only. This was a real failure in the prototype: the HTML version relied on a claude.ai-only proxy and died with a CORS error the moment it ran outside claude.ai. No Anthropic call may ever originate from the browser.

File parsing for resume uploads (PDF and DOCX to text) happens server-side. Use pdf-parse or similar for PDF and mammoth for DOCX. The prototype parsed in-browser with pdf.js and it worked, but server-side parsing is more reliable and keeps the client thin. After extraction, show the user the extracted text for review before saving, because PDF layouts sometimes garble; the prototype's pattern of showing a character count and asking the user to review the text is worth keeping.

## Data model

A user has one or more profiles (usually one: themselves, but allow more so someone can help a friend). A profile has: a name, one or more resume versions (label plus full text; multiple versions are a first-class concept, see below), an optional known-gaps text (each gap ideally with an honest mitigation), optional style notes (spelling preference, letter voice), a set of custom document formats (name plus plain-language format instructions), and a list of preferred job boards. Analyses, generated documents, and interview preps should be persisted per profile with the JD they ran against, timestamped, so a user can revisit an application later. That history table is also the seed of a future application tracker, which is deliberately out of scope for v1 but should not be structurally blocked.

Multiple resume versions matter because real candidates maintain several (the owner has three, each emphasizing a different side of his background). The union of all versions is the source of truth for fit analysis. The merged text sent to the model labels each version, tells the model that differing phrasings across versions are emphasis rather than contradiction, and tells it to flag genuine factual contradictions (dates, titles) rather than silently picking one. The fit analysis should also recommend which version is the best base for tailoring toward the specific role. This logic exists in the prototype and worked well; port it.

## The four stages

Stage 0, job finder. Server route that calls the Anthropic API with web search enabled, steered toward the profile's preferred boards plus any the user adds (adding a board just adds it to search targeting; be honest in the UI that this is search steering, not a deep integration). The prompt requires that every returned URL come from an actual search result, never constructed from memory, and asks for an honest caveat line about coverage. Results render as cards with title, company, location, fit band (high, medium, stretch), a two-sentence why grounded in the candidate's real experience, and a link out. The user opens a posting, copies the JD, and pastes it into Stage 1. Do not promise scraping or authenticated portal access; the boards prohibit it and it will break.

Stage 1 and 2, JD intake and fit analysis. A pasted JD (minimum length guard around 200 characters) goes to a fit-analysis route. The prompt asks for: the real role in one sentence versus its title (titles mislead; the reference JD in this repo is titled VP Product Development but describes hands-on engineering leadership), a role archetype, hidden signals (repeated words, oddly specific tool mentions, leftover reviewer comments, internal jargon; these reveal what the hiring team actually cares about), a verdict (apply confidently, apply with positioning work, or skip) with honest reasoning, a suggested interview prep format, the best base resume version when multiple exist, and 8 to 12 requirements each scored strong, adjacent, partial, or gap, with specific evidence including numbers, and a mitigation line for every partial or gap. Render as a verdict card, a hidden-signals callout, and a requirements ledger table.

Stage 3, documents. Format system with two built-ins and user-defined custom formats. ATS plain text: caps section headers, reverse chronological, no tables or columns or special characters, plain hyphen bullets, JD keywords woven in only where honest, dates as Mon YYYY. Standard formatted: polished conventional structure. Custom formats are plain-language instructions the user writes (for example, "each role max 2 lines, education first, cover letter max 200 words") that get injected into the generation prompt; they are stored on the profile. Output includes the complete resume, the cover letter, and a before-and-after list of retargeted bullets with a one-line why per bullet. Cover letter rules that worked: 250 to 350 words unless the format says otherwise, a company-specific hook in sentence one, a proof paragraph with numbers, a differentiator paragraph, a confident close, and zero sentences that could appear in another candidate's letter. Provide copy and download actions, and a live preview pane (monospace rendering for ATS, light typographic rendering for other formats). Since the generated text is format-shaped, switching formats triggers regeneration, not a re-render; the prototype made this explicit to the user and it felt honest rather than slow.

Stage 4, interview prep. Six formats, each with its own prep structure baked into the prompt: consulting (case openers with three-step structures, market sizing in the role's domain, fit/PEI stories, case math reminders), product management (product sense grounded in the company's actual product, execution and metrics questions, strategy, technical comfort probes, behavioral), investment banking (valuation technicals, three-statement walks, merger math, market awareness, motivation), VC/PE (deal judgment with a thesis skeleton from real diligence experience, market theses, term sheet basics, portfolio scenarios), tech leadership (system design for the JD's actual stack, technical depth spot checks, people and delivery scenarios, gap probes hit hardest), and general behavioral (classic categories each mapped to a distinct story so stories are not overused). Every prep output includes format notes on how the interview runs and what is scored, 8 to 10 likely questions with skeleton answers for the top 5 or 6 drawn from real profile stories with their numbers, gap defenses (acknowledge, adjacent evidence, learning velocity, three sentences each), practice drills, and reverse questions derived from the JD's hidden signals. The fit analysis pre-selects the suggested format; the user can override.

## Style rules for all generated text

These are non-negotiable and belong in a shared prompt constant: never use em dashes anywhere; no AI filler (delve, thrilled, excited to apply, seamlessly, cutting-edge unless quoting a JD, leverage as filler); confident, specific, understated, numbers do the boasting; active voice, short sentences; resume bullets are action verb plus what plus quantified outcome, and a metric is never dropped in rewording; never fabricate. Respect the profile's own style notes (the owner's profile uses British/Singapore spelling to match his existing resume; other users may differ).

## Prototype lessons worth keeping

The JSON-response pattern from the prototype was fragile in an instructive way: prompts demand JSON-only responses, and the client still defensively extracts the first-to-last brace span before parsing, because models occasionally wrap output anyway. In the hosted app, use the SDK's structured output support if available, or keep the defensive parse. Every generation route needs a retry-visible error state; the prototype surfaced failures with a retry button rather than a blank screen. Switching the active profile must reset all downstream stage state so nothing leaks between people. Escape all model-generated and user-provided text before rendering; the prototype's test suite specifically verified injection safety on the preview renderer, and the hosted app should carry equivalent tests.

## Reference material in this repo

The prototype HTML file (jd-career-suite-tool.html) is the working reference for all prompts, the format definitions, the preview renderers, and the UX flow; its final version passed a 31-test logic suite covering profile normalization, multi-resume merging, board management, custom formats, preview escaping, export shape, and input guards. The Claude Skill (jd-career-suite.skill) contains the reasoning methodology in prose form: the four-stage workflow, the owner's master profile with claimable facts and known gaps, and the writing style rules. The skill's master-profile file is the seed data for the owner's profile in the app; import it rather than retyping.

## Out of scope for v1

Application tracking dashboards, email or LinkedIn outreach automation, payment or billing, public signup, mobile apps, and scraping or authenticated integrations with job portals. The data model should leave room for the tracker, since analyses are already persisted per JD.

## Definition of done for v1

A classmate can be invited, sign in, create a profile by uploading two resume versions (one PDF, one DOCX) and reviewing the extracted text, run the job finder and get real linked postings with an honest coverage caveat, paste a JD and get a fit verdict with a requirements ledger, generate an ATS resume and cover letter plus one custom format they defined themselves, build interview prep in two different formats and see genuinely different structures, and log out and back in to find everything still there, visible only to them.
